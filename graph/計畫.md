## ⚠️ 環境隔離原則（最優先，所有 phase 都要遵守）

本計畫的**所有檔案、所有套件、所有產物**都必須限制在專案資料夾 `graph/` 之內。執行此計畫時請嚴格遵守：

### 不可以做的事
- ❌ **不可** `pip install` 到全域 / 系統 Python（必須先建 venv）
- ❌ **不可**修改 PATH、環境變數（machine / user 範圍）、系統設定、登錄檔
- ❌ **不可**修改 `C:\Users\user\` 底下任何專案外的設定（如 `.bashrc`、`.condarc`、`pip.ini`、`AppData` 系統檔）

- ❌ **不可**寫檔到 `graph/` 之外（除了 venv 自己會建立的 site-packages，那不算）

## Context

作者正在做鐵人賽 30 天「從 RAG 到 EmoRAG」系列，第 23 天介紹了 GraphRAG 的概念但只停留在理論層面。已熟悉傳統 vector RAG，想透過**親手實作**理解：

1. **每個環節**在做什麼（chunking → entity 抽取 → 子圖合併 → 社群偵測 → 報告 → 檢索）
2. **演算法迭代**的差異（特別是 **Louvain vs Leiden**——文章只講歷史，沒看到差異）
3. **三種檢索**（Local / Global / Drift）的實際行為差異
4. **產出可看懂的圖譜**——明確擔心做出「毛球圖」看不懂

**關鍵設計決策**：

- **語言**：Python 3.10+
- **路線**：從零自刻，不用微軟 GraphRAG 套件——每行程式碼都看得到、改得了
- **LLM 角色**：**不呼叫外部 API**，由執行此計畫的 Claude Code session 透過 Agent subagent 在 session 內完成所有 LLM 任務（entity 抽取、關係抽取、社群摘要）。每階段邊做邊解釋。
  - 注意：搬到其他電腦時，依然需要在該電腦上開啟 Claude Code session 來跑這個計畫。如果改用 API key 路線，需把 entity 抽取與摘要那兩步改成呼叫 Anthropic / OpenAI / Gemini API。
- **語料**：`論文20/` 共 173 個 PDF（營養 / 公衛 / 全球疾病負擔領域）
  - 根目錄 8 個中文 PDF 是作者整理的「主題摘要」
  - 子資料夾是各主題對應的英文原始論文（The Lancet、Current Developments in Nutrition 等）
  - **分階段語料策略**（避免一次處理 173 PDF 拖死流程）：
    - **Stage A（學習用）**：只用根目錄 8 個中文摘要 PDF → 預計 ~30–60 chunks，1–2 小時跑完，圖譜社群結構明顯
    - **Stage B（中規模）**：加入 3–5 個主題子資料夾的英文原始論文 → ~200–400 chunks
    - **Stage C（完整）**：全部 173 PDF → 留到流程穩定後再跑，預期數小時
  - **預期社群**（直覺判斷主題群）：心血管/油脂、母嬰營養、微量營養素缺乏、碳水化合物與死亡率、全球疾病負擔、永續飲食
- **社群偵測**：**同時跑 Louvain 和 Leiden** 並排比較，這是學習重點

## 實作階段（10 步驟，邊做邊解釋）

### Phase 0：環境準備 + 語料探勘
- 建 venv、寫 `requirements.txt`、`pip install`
- 確認 sentence-transformers 模型能下載（首次約 300MB）
- 建立目錄樹
- **PDF 解析**：用 `pypdf` 或 `pdfplumber` 把 Stage A 的 8 個中文 PDF 轉成純文字（注意中文 PDF 常見問題：直排、表格、頁眉頁尾雜訊），存到 `corpus_text/*.txt`
- 印出每份文件的字元數讓使用者確認解析品質

### Phase 1：Chunking（`01_chunk.py`）
- **做什麼**：依文章規格——1200 token、overlap 100，切中文文本
- **要解釋**：為什麼要 overlap？中文 token 化怎麼做？（用 `tiktoken` cl100k_base 估算）
- **產出**：`data/chunks.jsonl`（每行 `{id, text, char_range}`）
- **看得到**：`notes/step_01_chunking.md` 列前 3 個 chunk 的長度與內容

### Phase 2：Entity & Relation Extraction（`02_extract_entities.py` + Claude）
- **做什麼**：對每個 chunk，**Claude 自己讀**，抽出 entities + relations + 每個關係 1–10 強度分數
- **流程**：
  1. Python 把 chunks 整理成 prompt（含中文 entity types：人物/地點/組織/事件/概念）
  2. **Claude** 對每個 chunk 直接抽取，寫入 `data/entities.jsonl` 和 `data/relations.jsonl`
  3. 小語料（<10 chunks）順序做；大語料用 Agent 並行多個 subagent
- **要解釋**：為什麼要規定 entity type？為什麼要強度分數？正規化（同義詞合併）為何重要
- **產出**：兩個 JSONL + `notes/step_02_extraction.md` 展示一個 chunk 的完整抽取結果

### Phase 3：子圖合併（`03_merge_subgraphs.py`）
- **做什麼**：跨 chunk 合併同名實體；同 source-target 關係合併；描述匯總成 array 後（再請 Claude）摘成單一描述
- **要解釋**：「實體解析」(entity resolution) 為什麼是 GraphRAG 的關鍵痛點；大小寫/全半形/簡繁體會怎麼壞
- **產出**：`data/graph.gml`（NetworkX 標準格式，可被 Gephi 讀）

### Phase 4：原始圖譜視覺化 v1（`04_visualize_raw.py`）
- **做什麼**：用 pyvis 產出互動 HTML，節點大小 = degree，邊粗細 = 強度分數
- **要解釋**：這就是「毛球圖」的原型——還沒社群著色前，看起來就是一團
- **產出**：`viz/04_raw_graph.html`（用瀏覽器開）

### Phase 5：節點嵌入（`05_embed_nodes.py`）
- **做什麼**：用 sentence-transformers 對每個 entity 的「name + description」嵌入，存入 ChromaDB
- **要解釋**：為什麼不直接嵌 chunk？entity-level embedding 對 Local Search 的意義
- **產出**：本地 ChromaDB collection `entities`

### Phase 6：社群偵測——**Louvain vs Leiden 並排對照**（`06a/b/c`）
這是整個專案的學習高潮，對應「想看出算法差異」的需求。

#### 6a. Louvain
- 跑 `python-louvain` 的 `community.best_partition`
- 印出：社群數、modularity 分數、每個社群大小、**檢查每個社群是否連通**（用 NetworkX `is_connected`）
- **預期會看到**：可能出現「不連通的社群」——這正是 Leiden 論文要解決的問題！

#### 6b. Leiden
- 跑 `leidenalg.find_partition` 用 `RBConfigurationVertexPartition`（CPM）
- 印同樣指標
- **預期看到**：modularity 略高、社群保證連通、執行更快

#### 6c. 對照與視覺化
- pyvis 雙視窗：左邊 Louvain 著色、右邊 Leiden 著色
- 統計表：方法 / 社群數 / modularity / 是否全部連通 / 執行時間
- **要解釋**：「不連通社群」的具體案例——指出某個社群裡面其實是兩個分離的小團
- **產出**：`viz/06_louvain_vs_leiden.html` + `notes/step_06_algorithm_comparison.md`

### Phase 7：Community Reports（`07_community_reports.py` + Claude）
- **做什麼**：對每個 Leiden 社群（含其節點+關係+原始 chunk），**Claude** 寫一份結構化摘要：標題、概要、關鍵實體、關鍵發現
- **要解釋**：為什麼這步是 GraphRAG 區別於傳統 RAG 的關鍵（提供 Global Search 的彈藥）
- **產出**：`data/reports.jsonl` + 把每份報告也嵌入到 ChromaDB collection `reports`

### Phase 8：三種檢索（`08a/b/c`）
針對同一個查詢，分別跑三種檢索，**輸出每一步的中間結果**讓使用者看到差異。

#### 8a. Local Search（細節型問題）
- Query → 嵌入 → 在 entity collection 找 top-k → 圖遍歷取 1-hop 鄰居 + 對應 chunks + 所屬社群報告 → 組成 context → Claude 回答
- **追蹤檔**：`viz/08a_local_trace.html` 視覺化「檢索路徑」（哪些節點被啟動、走了哪些邊）

#### 8b. Global Search（廣泛主題）
- Query → 嵌入 → 在 reports collection 找 top-k → Claude 從每份報告抽 key points + 評分 → Map-Reduce 聚合 → 最終回答

#### 8c. Drift Search（兼顧廣深）
- HDE（先讓 Claude 寫個假設答案）→ 嵌入這個假設文件 → 找 reports → 生成初答 + 後續問題 → 對每個後續問題切到 Local Search → 迭代

### Phase 9：對照組——傳統 Vector RAG（`09_baseline_vector_rag.py`）
- 同樣的 chunks，純向量檢索 + Claude 回答
- 用同樣 3 個查詢（一個細節型、一個主題型、一個多跳推理型）跑全部 4 種方法
- **產出**：`notes/step_09_comparison.md` 結果表格

## 關鍵設計：避免「毛球圖」

每個視覺化階段都會做這些事：

1. **節點上限**：超過 200 個節點時自動篩選 top-degree
2. **社群著色**：Phase 6 之後一律按社群上色
3. **邊過濾**：強度 < 3 的邊預設隱藏（可切換）
4. **階層展開**：pyvis `hierarchical layout` 讓社群分區站開
5. **hover 顯示**：滑鼠移到節點顯示完整描述、所屬社群、degree

## 驗證方式（end-to-end）

完成後逐項檢查：

1. **Phase 1–3**：`graph.gml` 在 Gephi/Cytoscape 可開，節點數合理
2. **Phase 4**：瀏覽器打開 `viz/04_raw_graph.html` 可拖曳互動
3. **Phase 6**：對照表清楚顯示 Louvain 在你的語料上是否真的出現「不連通社群」（這是論文預測的現象，能複現就達成核心學習目標）
4. **Phase 7**：人工檢視 1–2 份 community report，確認摘要合理
5. **Phase 8**：對「人物 X 與事件 Y 的關係」這種需多跳推理的問題，Local/Global/Drift 應該明顯優於 Phase 9 的 baseline
6. **Phase 9**：把四種方法的回答放在同一份 markdown 並排，看出差異

   python -m venv .venv

## A.1 起點：作者的擔憂

**作者問**：「我想實作一個 graph rag 的實作，但我擔心最後出來的圖譜我會看不懂…」

這是學習 GraphRAG 最常見的擔憂——做完之後是一團「毛球」，看不出主題、看不出社群。

**Claude 的初步分析**：圖看不懂通常源自三個原因：
1. **節點太多太密**：LLM 抽出的 entities 沒過濾，幾百個節點擠在一起
2. **沒有層次**：全部平鋪、看不出哪些是主題、哪些是細節
3. **標籤雜亂**：LLM 抽出的實體名稱沒正規化（「久石讓」「Joe Hisashi」「久石讓(作曲家)」變三個節點）

**Claude 給的三條路線比較**：

| 方案 | 工具 | 視覺化友善度 | 適合場景 |
|---|---|---|---|
| A. 微軟 GraphRAG | `graphrag` 套件 | ⭐⭐ (要自己接 Gephi) | 想完整體驗論文方法 |
| B. LlamaIndex KG | `KnowledgeGraphIndex` | ⭐⭐⭐⭐ (內建 pyvis HTML) | 快速 demo、可互動 |
| C. Neo4j + LLM Graph Builder | Neo4j Desktop | ⭐⭐⭐⭐⭐ (官方 UI) | 想真的「看得懂」 |

→ Claude 起初推薦 C（Neo4j 看得最清楚）。

## A.2 文章整理：iThome 第 23 天 GraphRAG

作者貼了自己的 iThome 鐵人賽文章 [連結](https://ithelp.ithome.com.tw/articles/10394746)，要 Claude 整理。文章核心內容（整理後）：

### 知識圖譜三要素
| 要素 | 圖論對應 | 說明 |
|---|---|---|
| 實體 Entity | 節點 Node | 從文本塊抽出的人/事/物/概念 |
| 關係 Relationship | 邊 Edge | 連接兩節點，含 1–10 強度分數 |
| 社群 Community | 節點集群 | 用 Leiden 等演算法分層偵測 |

### Microsoft GraphRAG 索引流程
```
非結構化文本
   ↓ (1) 切塊：1200 token / overlap 100
文本塊 chunks
   ↓ (2) LLM 抽 entities + relationships（給 1–10 強度分）
子圖 subgraphs
   ↓ (3) 同名同類型實體合併、彙總描述
合併後的全域圖
   ↓ (4) 節點嵌入（Node2Vec）+ 社群偵測
分層社群
   ↓ (5) LLM 對每個社群寫摘要 → Community Reports
   ↓ Community Reports 也丟進向量庫
```

### 三種檢索
| 類型 | 適合問題 | 核心流程 |
|---|---|---|
| Local Search | 細節型 | 語義搜 entity → 圖遍歷 → 抓相連 chunks/reports/relations |
| Global Search | 主題型 | 用 community reports 檢索 → LLM 抽 key points → Map-Reduce |
| Drift Search | 兼具廣深 | HDE 假設文件 → 找 reports → 生成初答 + 後續問題 → 切到 Local Search 迭代 |

### Louvain → Leiden 演算法演進
- **Modularity 最大化** 是社群偵測主流目標 → NP-hard
- **Louvain**（比利時 Louvain 大學命名）：最熱門啟發式
- **Louvain 致命缺陷**：可能產生「內部不連通」的社群（與 resolution limit 無關）
- **Leiden 改進**：保證社群內部連通、收斂到穩定局部最優、比 Louvain 更快、Smart Local Move + Fast Local Move + Random Neighbour Move

### Raptor（長上下文 RAG）
1. 文件層級嵌入（不切塊）
2. 叢集分組 + 高層次摘要
3. 遞迴形成文件樹

### 文章中沒提但實作會踩到的缺口
- 沒有任何程式碼範例
- 沒講中文支援（Microsoft GraphRAG 預設 prompt 是英文）
- 沒講成本（entity extraction 階段大量呼叫 LLM）
- 沒講視覺化怎麼做（作者擔憂的核心問題）
- Leiden 只講歷史、不講參數

## A.4 為什麼這個計畫長這樣（設計理念）

從上述對話可以看出計畫的三條主軸：

### 主軸一：「學會」優於「跑得通」
- 從零自刻而非用現成套件 → 每個環節都看得到、改得了
- Louvain vs Leiden 並排對照 → 親眼看到論文預測的「不連通社群」現象
- 每個 phase 都產出 `notes/step_XX_*.md` → 可直接拿去寫鐵人賽
- 加 Phase 9 vector RAG 對照組 → 看出 GraphRAG 真正的價值在哪

### 主軸二：「看得懂」優於「做得多」
- 視覺化貫穿全程（Phase 4 原始圖、Phase 6 社群著色、Phase 8 檢索路徑）
- 5 條防毛球規則（節點上限、社群著色、邊過濾、階層 layout、hover）
- 分 Stage A/B/C → Stage A 30–60 chunks 圖譜清晰，先建立直覺再放大

### 主軸三：「乾淨」優於「方便」
- 環境隔離原則放在文件最頂端
- 所有產物都在 `graph/` 內，刪資料夾即完全復原
- HF 快取重定向到專案內，避免污染 `~/.cache/`

## A.5 給未來 Claude session 的指示（新電腦接手時）

當這份計畫在另一台電腦上執行時，新 Claude 應該：

1. **先讀完整份 `計畫.md`**（含本附錄），理解前因後果
2. **遵守「⚠️ 環境隔離原則」**，所有操作侷限於 `graph/`
3. **從 Phase 0 開始按順序執行**，每階段：
   - 跑出產物（`data/`、`viz/`、`notes/`）
   - 寫 `notes/step_XX_*.md` 用作者的角度說明這步在做什麼、為什麼
   - 停下來等作者確認，再進入下一階段
4. **遇到要動專案外的事**，先停下來說明 + 給清除指令 + 取得同意
5. **不要跳過 Phase 6 的對照分析**——這是整個學習計畫的核心高潮
6. **語料先跑 Stage A（8 個中文 PDF）**，跑通後再問作者要不要擴大到 B / C

# 附錄 B：思維鏈與取捨（為什麼選這個、不選那個）

每一個技術選擇背後都有**被淘汰的選項**。這段把思考過程記錄下來，避免之後忘記為什麼這樣選、或在新電腦上盲目改。

## B.1 圖結構：NetworkX vs igraph vs Neo4j

| 候選 | 優點 | 缺點 | 結論 |
|---|---|---|---|
| **NetworkX**（選） | 純 Python、API 直觀、Gephi 可讀其 GML | 大圖（>100k 節點）會慢 | ✅ 教學首選 |
| igraph | 速度快（C 底層） | API 沒那麼直觀、文件較少 | 拿來跑 Leiden 就好 |
| Neo4j | 視覺化最佳、Cypher 強大 | 要裝 Desktop（>500MB）、違反「環境隔離原則」 | ❌ 違反隔離 |

**取捨**：作者擔心毛球圖 → 直覺會選 Neo4j，但 Neo4j 要裝桌面程式、佔系統資源、難拆裝。改用 **NetworkX 建圖 + pyvis 互動 HTML 視覺化** 達到 80% 效果但 0% 系統污染。如果未來真要 Neo4j，可以從 `graph.gml` 匯入。

## B.2 社群偵測：兩個套件並用

**Louvain → `python-louvain`**，**Leiden → `igraph` + `leidenalg`**

為什麼不只用 igraph 跑兩種？因為：
1. `python-louvain` 是純 Python 實作，更貼近原始論文，容易讀原始碼
2. 兩個套件各跑自己 → 結果差異更可信（避免「同一作者實作偏好同方向」）
3. 對學習來說，「兩個獨立實作得到差不多的 modularity」比「同套件兩個 function」更有說服力

## B.3 嵌入：本地 vs API

| 候選 | 成本 | 中文能力 | 隔離性 |
|---|---|---|---|
| **sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2**（選） | 0 | 中等（50 種語言） | ✅ 模型可放專案內 |
| OpenAI text-embedding-3-small | $$ | 高 | ❌ 要 API key、要連外 |
| BGE-M3（中文最強） | 0 | 極高 | ✅ 但模型 2GB，太重 |

**取捨**：MiniLM 是 best-of-trade-off。300MB 可放 `graph/.cache/`，效果對 Stage A 學習用足夠。如果後期發現中文嵌入太弱（例如 entity 同義詞合不到一起），再升級到 BGE-M3。

## B.4 向量庫：ChromaDB vs FAISS vs 純 NumPy

| 候選 | 優點 | 缺點 | 結論 |
|---|---|---|---|
| **ChromaDB**（選） | 本地檔案、API 簡單、有 metadata 篩選 | 寫入比 FAISS 慢一點 | ✅ 教學夠用 |
| FAISS | 最快、最省記憶體 | 沒 metadata、要自己包 | 規模 >100k 才有意義 |
| 純 NumPy + cosine | 最直觀，看得到內部 | 大規模慢 | 可作 Phase 9 baseline 教學 |

**取捨**：教學重點不在於最快搜尋，而在於「能存、能查、能改」。ChromaDB 把資料庫存成檔案放 `graph/chroma_db/`，刪資料夾就清掉。

## B.5 視覺化：pyvis vs Gephi vs Cytoscape

| 候選 | 互動性 | 安裝負擔 | 結論 |
|---|---|---|---|
| **pyvis**（選） | HTML，瀏覽器可拖、可縮放、hover 顯示 | 純 pip 套件 | ✅ 主力 |
| Gephi | 最強佈局演算法 | 要裝 Java + GUI | 選用，作為 GML 後備檢視 |
| Cytoscape | 生物資訊圈標準 | 要裝 GUI | 不必 |

**取捨**：產生 `viz/*.html` 用瀏覽器即可看，不污染電腦。`graph.gml` 同時保存，未來想用 Gephi 仍可。

## B.6 PDF 解析：pypdf vs pdfplumber vs PyMuPDF

- **pypdf**：純 Python、輕量、無系統依賴（首選）
- **pdfplumber**：表格處理較好（fallback）
- **PyMuPDF (fitz)**：解析最強但要 C 二進位、AGPL 授權有疑慮 → 避免

中文 PDF 直排、雙欄、表格雜訊本來就難搞。Phase 0 要印字數讓作者確認解析品質，必要時改用 OCR（但會破壞隔離原則，要先停下來討論）。

## B.7 為什麼不用 Microsoft GraphRAG 套件

被淘汰的理由：
1. **黑盒**：很多細節包在 prompt template 和 yaml 設定裡，看不到、改不動
2. **預設英文**：中文要改 prompt 和 settings，反而比自己刻還累
3. **重**：依賴 OpenAI/Azure，違反「不用 API key」決策
4. **學不到**：作者明確說想看到「算法迭代差異」——用套件就被遮蔽了

但**會在 Phase 9 之後可選地**讀微軟原始碼對照學習（不安裝、只讀 GitHub）。

## B.8 為什麼分 Stage A/B/C 而不直接跑全部

- Stage A 8 PDF → ~30–60 chunks → ~200–400 entities：圖譜剛好「看得清楚但不單薄」
- 全部 173 PDF → ~5000+ chunks → ~10000+ entities：第一次跑就掉進毛球地獄、且抽 entity 要花一整天
- 學習的關鍵在於 **每個階段都能停下來檢查產物**，量太大失去這個優勢

## B.9 為什麼用 Claude session 內 Agent 而不是 API

| 路線 | 成本 | 解釋性 | 可攜性 |
|---|---|---|---|
| **Claude Code Agent**（選） | $0（已付 Code 訂閱） | 邊做邊講 | 新電腦也要裝 Claude Code |
| Anthropic API | $1–10/Stage A | 要看 log | API key 要帶 |
| Ollama 本地 | $0 | 慢、品質差 | 模型檔大（4GB+） |

**取捨**：作者希望「邊做邊解釋」→ Claude Code 是天然契合。代價是新電腦也要登入 Claude Code（已寫進「跨電腦執行 checklist」）。

## B.10 為什麼計畫文件這麼長

短計畫的問題：執行時細節有歧義、忘記取捨、新電腦的 Claude 重新問一遍。
長計畫的代價：作者要先讀。
**選擇長計畫**，因為這是學習專案、可以慢慢消化；且新電腦的 Claude 讀完就能無縫接手。
